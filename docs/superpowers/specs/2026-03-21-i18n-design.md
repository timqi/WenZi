# I18n Design Spec — Chinese + English Support

## Overview

Add internationalization (i18n) support to WenZi, starting with Chinese and English. The app name displays as "WenZi" in English and "闻字" in Chinese.

## Goals

- Support Chinese (`zh`) and English (`en`) UI
- Language selection: config setting > system locale > English fallback
- Allow user to choose language in Settings or follow system default
- Zero external dependencies — pure Python + JSON translation files
- Future-extensible: adding a language means adding one JSON file

## Non-Goals

- Real-time language switching without restart
- Translating log messages, LLM prompts, or internal error messages
- Community translation tooling (gettext, Crowdin, etc.)

---

## Architecture

### File Structure

```
src/wenzi/
├── i18n.py              # Core translation module
└── locales/
    ├── __init__.py
    ├── en.json           # English (default/fallback)
    └── zh.json           # Chinese
```

### Core API — `i18n.py`

```python
# Get translated string
t("menu.quit")                          # → "Quit" or "退出"
t("alert.update.restart_title", version="1.2")  # → "Restart to update 1.2?"

# Set/get current locale
set_locale("zh")
get_locale()            # → "zh"

# Initialize (called once at app startup)
init_i18n(locale=None)  # None → read from config → system locale → "en"
```

### Translation Lookup

1. Look up key in current locale's JSON
2. Key not found → fallback to `en.json`
3. Still not found → return the key itself (helps catch missing translations during development)

### Parameter Interpolation

Uses Python `str.format_map()`:

```json
{ "alert.update.restart_title": "Restart to update v{version}?" }
```

```python
t("alert.update.restart_title", version="1.2")  # → "Restart to update v1.2?"
```

## Configuration

### New Config Field

Add `language` to the existing JSON config file:

```json
{
  "language": "auto"
}
```

Valid values: `"auto"` | `"en"` | `"zh"`

### Locale Resolution at Startup

```
read config.language
  → "auto" → detect system locale via locale.getlocale()
    → starts with "zh" → "zh"
    → otherwise → "en"
  → "en" or "zh" → use directly
```

## Settings UI — Language Dropdown

Add a language dropdown to the **General** tab in Settings, positioned at the top (language is the most fundamental preference).

**Options:**

| Display Text | Config Value |
|-------------|-------------|
| System Default | `"auto"` |
| English | `"en"` |
| 中文 | `"zh"` |

**On change behavior:**

1. Save to config
2. Show alert: "Language change will take effect after restart" / "语言更改将在重启后生效"
3. No dynamic refresh — restart required

## WKWebView Components

Web-based UI components (`result_window_web.py`, `chooser_html.py`, `history_browser_window_web.py`) cannot call Python's `t()` directly.

### Solution: JS Translation Injection

**Python side** — inject translations when showing the panel:

```python
translations = get_translations_for_prefix("preview.")
script = f"window._i18n = {json.dumps(translations)};"
webview.evaluateJavaScript_completionHandler_(script, None)
```

**JS side** — helper function in HTML templates:

```javascript
function i18n(key) {
    return window._i18n?.[key] ?? key;
}
```

`get_translations_for_prefix(prefix)` returns a subset of the current locale's translations matching the given prefix, with the prefix stripped from keys for cleaner JS usage.

## String Replacement Strategy

### What to Replace

| UI Type | Location | How |
|---------|----------|-----|
| Menu items | `app.py`, `menu_builder.py` | Replace title strings with `t()` |
| Settings labels/buttons | `settings_window.py` | Replace `setTitle_()` / `setStringValue_()` args |
| Alerts/dialogs | All controllers | Replace `_topmost_alert()` title/message args |
| Notifications | Controller call sites | Replace `send_notification()` title/subtitle args |
| Status bar text | `app.py` | Replace status strings ("Recording...", etc.) |
| WKWebView HTML | `result_window_web.py`, `chooser_html.py` | JS injection + `i18n()` function |
| Config labels | `config.py` | Replace display names (MODIFIER_KEY_CHOICES etc.) |

### What NOT to Replace

- Log messages (for developers, stay in English)
- LLM prompt templates (for AI processing)
- Internal error messages (not user-facing)
- Variable names, config keys, enum values

## Testing

### Unit Tests — `test_i18n.py`

- `t()` returns correct value for `en` and `zh`
- Parameter interpolation works: `t("key", name="x")`
- Missing key in `zh` falls back to `en`
- Missing key in both returns the key itself
- `set_locale()` / `get_locale()` work correctly
- `init_i18n()` priority: config > system locale > "en"
- `get_translations_for_prefix()` returns correct subset

### Completeness Test — `test_locale_completeness.py`

- Load `en.json` and `zh.json`
- Assert both have identical key sets
- CI fails if any translation is missing

### Existing Tests

- Update any string-matching assertions that reference hardcoded UI text
- Expected impact: minimal

### Not Tested

- WKWebView JS-side `i18n()` function (requires real WebKit runtime)
- Settings UI language dropdown interaction (PyObjC UI tests are fragile)
