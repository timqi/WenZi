前置条件

  • 已开启预览模式（设置 → 通用 → Preview）
  • 已配置至少一个 LLM 供应商（设置 → LLM）
  • 已选择一个增强模式（设置 → AI）

核心优化工具

1. Play ▶ — 确认你到底说了什么

在归咎 ASR 或 AI 之前，先回放录音。这能帮你判断问题出在哪个层级：

听到的情况 问题诊断 应对措施
自己说错了或口齿不清 源头问题 重新录制，说清楚一些
说得没问题，但 ASR 文本有误 ASR 问题 切换 ASR 引擎、开启 Punc、或尝试其他模型
ASR 文本正确，但 AI 输出有误 提示词问题 检查并迭代优化提示词（见下文）

2. Prompt 按钮 — 查看实际发送的系统提示词

增强完成后点击 Prompt ⓘ，查看发送给 LLM 的完整系统提示词。这是优化的最关键工具 — 看不到提示词，就无法改进它。

重点关注：

  • 指令是否足够清晰和具体？
  • 是否提到输入来自 ASR（可能包含识别错误）？
  • 是否有"仅输出结果"的规则防止 AI 添加多余说明？
  • 是否覆盖了边界情况？

3. 模式切换 (⌘1–⌘9) — 即时对比不同模式

按 ⌘1 到 ⌘9 可以不重新录音直接切换模式。结果会被缓存 — 切回之前的模式会瞬间显示缓存结果（标记 [cached]）。

用途：

  • 对比不同提示词处理同一段语音的效果
  • 对新模式和已有模式进行 A/B 测试
  • 找出最适合特定内容类型的模式

4. LLM 模型下拉框 — 对比不同模型

通过 AI 下拉框切换模型，查看同一提示词在不同 LLM 上的表现。每个（模式、模型、thinking）组合的缓存是独立的。

5. Thinking 模式 — 观察 AI 推理过程

勾选 thinking 复选框，增强完成后点击 🧠（大脑）按钮查看 AI 的推理过程。可以揭示：

  • AI 为什么做出特定的用词选择
  • AI 在哪些地方感到困惑或不确定
  • AI 是否正确理解了你的意图

这对诊断提示词问题极为重要 — 如果 AI 的推理逻辑就是错的，说明提示词需要进一步澄清。

6. Final Result — 修正并训练

在确认前编辑 Final Result 修正任何遗留问题。每次编辑都会被标记为 user_corrected，逐步积累到词库系统中。

分步优化流程

第一阶段：识别问题

  1. 录制一句有代表性的语音。
  2. Play ▶ 回放录音，确认你说的和你想的一致。
  3. 检查 ASR 文本 — 转写是否准确？
  4. 检查 AI 结果 — 增强改善了文本还是改坏了？
  5. 点击 Prompt — 阅读产生此结果的系统提示词。

第二阶段：诊断

利用面板控件缩小问题范围：

症状 诊断步骤 可能原因
AI 添加了多余的说明文字 检查 Prompt — 缺少"仅输出结果"规则 提示词需要加 "仅输出处理后的文本，不要添加说明"
AI 过度修正了本来正确的文本 检查 Prompt — 指令过于激进 提示词需要加 "正确的内容保持原样"
AI 误解专业术语 开启 thinking，查看 🧠 提示词需要补充领域上下文或示例
在某个模型上好用，换一个就不行 通过下拉框切换模型对比 提示词对特定模型依赖过强，需要更明确的规则
短文本好用，长文本出问题 分别用短句和长句测试 提示词需要考虑不同长度的输入

第三阶段：编辑模式

  1. 用文本编辑器打开 ~/.config/VoiceText/enhance_modes/<mode_id>.md。
    或使用 设置 → AI → 选择模式 → 编辑。
  2. 根据诊断结果做针对性修改。
  3. 重启应用（或重新加载配置）以加载更新后的提示词。
  4. 用同一句话重新录制，对比结果。

第四阶段：验证

  1. ⌘1–⌘9 在更新后的模式和其他模式间切换 — 更新后的是否更好？
  2. 切换 LLM 模型 — 改进在不同模型上是否都有效？
  3. 测试边界情况 — 尝试短输入、长输入、混合语言、嘈杂语音。
  4. 开启 thinking — AI 的推理过程是否符合你的预期？

第五阶段：持续改进

  • 坚持修正 Final Result — 词库系统会从你的编辑中学习。
  • 查看 History（点击 History 按钮）— 寻找 AI 反复出错的规律。
  • 构建词库（设置 → AI → Build Vocabulary）— 领域术语会自动积累。

实战案例

案例 1：AI 在翻译后附加了多余说明

问题："翻译为英文"模式输出了 "Translation: ..." 带有前缀。

诊断：点击 Prompt → 提示词只说了"翻译为英文"，但没有明确禁止附加说明。

修复：在提示词中添加：

仅输出翻译后的文本。
不要添加任何前缀、标签或说明。

案例 2：AI 错误地纠正了人名

问题：你说"找萍萍确认一下"，但 AI 改成了"找平平确认一下"。

诊断：开启 thinking → AI 因为缺少上下文，把"萍萍"当成了"平平"的 ASR 错误。

解决方案：

  • 在 Final Result 中改回"萍萍" → 纠正记录会积累并逐步构建词库
  • 开启对话历史（设置 → AI），让 AI 看到之前确认过的"萍萍"用法
  • 积累足够纠正后手动构建词库

案例 3：模式在某个模型上好用，换一个就不行

问题："命令行大神"在 GPT-4 上输出干净的 shell 命令，但在本地模型上会加上 markdown 代码块。

诊断：通过下拉框切换模型对比输出，点击 Prompt 检查提示词。

修复：让提示词更加明确：

仅输出原始命令。
不要使用 markdown 代码块或反引号包裹。
不要添加任何说明。

优化技巧

  • 每次只改一处。编辑提示词时，只修改一个地方然后测试。多处同时修改会导致无法判断哪个改动起了作用。
  • 保存好的测试用例。用 Save 导出能暴露提示词问题的录音。编辑提示词后重新播放验证修复效果。
  • 使用编号规则。LLM 对带编号规则的结构化提示词的遵循度，远高于段落式的自然语言描述。
  • 务必提及 ASR 上下文。在提示词中加入"用户输入来自 ASR，可能包含语音识别错误" — 这能显著提升纠错容错能力。
  • 关注 Token 计数。Tokens 显示（↑prompt ↓completion）帮你判断提示词效率。如果 ↑ 很高而 ↓ 很低，说明提示词可能过于冗长。
